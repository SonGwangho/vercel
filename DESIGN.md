# 디자인 시스템 명세: 디지털 큐레이터

## 1. 개요와 핵심 방향

홈 화면만 있는 쇼케이스가 아니라 다음과 같은 화면을 함께 품고 있다.

- 상단 탭 네비게이션
- 좌측 트리 메뉴 기반 정보 구조
- 게임 페이지
- 정보 페이지
- 위키형 문서 페이지
- 관리자 페이지

따라서 이 문서는 "화려한 한 화면"을 만드는 용도가 아니라, **현재 프로젝트 전반에 무리 없이 녹아드는 공통 디자인 기준**으로 사용한다.

핵심 원칙은 아래와 같다.

- 과한 장식보다 정보 구조를 우선한다.
- 여백, 배경 톤, 타이포 크기 차이로 계층을 만든다.
- 기존 프로젝트의 가벼운 사용성을 해치지 않는 범위에서 고급스러운 분위기를 더한다.
- 페이지 성격이 달라도 같은 서비스처럼 보이게 한다.

---

## 2. 현재 프로젝트에 맞춘 디자인 해석
### 2.1 전체 성격

- 홈 화면은 "요약 대시보드" 역할을 한다.
- 게임 화면은 조작과 가독성이 우선이다.
- 정보/위키 화면은 문서 탐색성과 계층 구조가 우선이다.
- 관리자 화면은 기능성, 안정감, 명확성이 우선이다.

즉, 모든 페이지를 동일하게 강한 아트 디렉션으로 밀어붙이지 않는다.
대신 아래 요소만 공통으로 유지한다.

- 정돈된 간격
- 선명한 제목 계층
- 과하지 않은 입체감
- 라이트/다크 모드 일관성
- 카드, 메뉴, 패널의 통일된 톤

### 2.2 현재 스타일과의 융화 원칙

현재 프로젝트는 이미 다음 특징을 가지고 있다.

- 라이트/다크 테마 전환 지원
- 상단 탭 메뉴와 좌측 사이드바 구조
- 비교적 밝고 단정한 카드형 UI
- `Pretendard`, `Noto Sans KR` 중심의 한글 친화적 본문 스타일

따라서 디자인 시스템 적용 시 다음을 지킨다.

- 본문 기본 글꼴은 기존처럼 `Pretendard`, `Noto Sans KR` 중심으로 유지한다.
- 영문 헤드라인, 숫자, 기술성 라벨에만 포인트용 서체를 제한적으로 사용한다.
- 기존의 너무 강한 테두리 사용은 줄이되, 현재 프로젝트의 명확한 사용성은 유지한다.
- 다크 모드는 "검은 배경 위 네온"보다 "차분한 차콜 계열"에 가깝게 정리한다.

---

## 3. 색상과 표면 철학

현재 프로젝트는 라이트/다크 모드를 모두 사용하므로, 이 문서는 특정 한쪽만 강제하지 않는다.
기본 원칙은 다음과 같다.

### 3.1 라이트 모드

- 기본 배경은 너무 새하얗지 않은 밝은 톤을 사용한다.
- 카드와 패널은 배경과 미묘하게 구분되는 정도의 표면색을 사용한다.
- 브랜드 포인트는 블루 계열을 유지한다.
- 경계선은 강한 회색 선보다 아주 옅은 색 차이로 처리한다.

권장 방향:

- 배경: 현재 `--bg` 계열 유지
- 기본 표면: 현재 `--surface` 계열 유지
- 강조 표면: 현재 `--surface-strong`를 조금 더 목적성 있게 사용
- 브랜드 컬러: 현재 `--brand`, `--brand-strong` 축 유지

### 3.2 다크 모드

- 완전한 순흑보다는 차콜 기반의 어두운 톤을 사용한다.
- 카드와 패널은 배경에서 한 단계 떠 있는 정도로만 구분한다.
- 텍스트 대비는 충분히 확보하되, 눈부신 순백은 남용하지 않는다.
- 강조색은 푸른 계열을 유지하되 넓은 면적보다 포인트에 집중한다.

### 3.3 선 대신 레이어 사용

이 프로젝트에서는 "무조건 선을 없앤다"가 아니라, **가능하면 선보다 배경 톤과 간격을 우선한다**로 해석한다.

- 카드, 패널, 메뉴 그룹은 먼저 배경색 차이로 나눈다.
- 그래도 구분이 부족하면 매우 옅은 경계선을 사용한다.
- 기존처럼 모든 블록에 뚜렷한 `1px` 선을 일괄 적용하지 않는다.

이 원칙은 현재 프로젝트와 특히 잘 맞는다.
이미 카드와 패널 구조가 많기 때문에, 선을 줄이고 표면 레이어를 정리하면 훨씬 덜 답답해진다.

## 4. 타이포그래피

현재 프로젝트는 한글 콘텐츠 비중이 높다.
따라서 기존 본문 글꼴 체계를 유지하면서, 강조 영역에만 개성을 더하는 방식이 적합하다.

### 4.1 기본 원칙

- **본문:** `Pretendard`, `Noto Sans KR`
- **영문 헤드라인 또는 강조 제목:** 필요 시 `Manrope`
- **숫자, 점수, 코드, 라벨:** 필요 시 `Space Grotesk`

### 4.2 적용 방식

- 일반 페이지 본문과 메뉴 텍스트는 기존 한글 친화 폰트를 유지한다.
- 홈 화면 주요 수치, 게임 점수, 랭킹 숫자, 기술성 메타 정보에만 `Space Grotesk`를 제한적으로 사용한다.
- 매우 큰 전시형 타이포는 홈 상단, 섹션 헤더, 위키 표지형 화면 등 일부에만 적용한다.

### 4.3 현재 프로젝트에서의 권장 계층

- 페이지 최상위 제목: 크고 명확하게
- 카드 제목: 중간 크기, 굵기 강조
- 설명 문구: 차분한 보조색
- 수치 정보: 본문보다 크게, 숫자 대비 강화
- 메뉴 텍스트: 과장 없이 명확하게

즉, 이 프로젝트에서는 "전부 에디토리얼"보다 **기능 중심 구조 위에 선택적으로 에디토리얼 감각을 얹는 방식**이 적절하다.

---

## 5. 깊이감과 입체 표현

이 프로젝트에 필요한 깊이감은 과한 그림자가 아니라, 패널 간 위계가 자연스럽게 느껴지는 수준이다.

### 5.1 권장 원칙

- 큰 레이아웃 구획은 배경 차이로 구분한다.
- 카드와 패널은 반경과 표면색으로 떠 보이게 한다.
- 그림자는 흐리고 약하게 사용한다.
- 다크 모드에서는 그림자보다 명도 차이와 반투명 배경이 더 효과적이다.

### 5.2 적용 대상

- 상단 바
- 사이드바
- 홈 카드
- 랭킹 카드
- 모달
- 드롭다운

### 5.3 주의점

- 위키나 정보 페이지는 장식보다 읽기 편한 구조가 더 중요하다.
- 관리자 페이지는 시각 효과보다 안정적인 정렬과 대비가 더 중요하다.

---

## 6. 레이아웃 기준

현재 프로젝트 구조를 기준으로 레이아웃 원칙을 정의한다.

### 6.1 상단 바

- 현재 상단 탭 구조는 유지한다.
- 브랜드 영역은 과도한 배지 느낌보다, 서비스의 정체성을 보여주는 정돈된 포인트로 다듬는다.
- 활성 메뉴는 배경색과 텍스트 강조로 명확히 표시한다.
- 상단 바는 가능한 한 선보다 표면 톤 차이로 구분한다.

### 6.2 사이드바

- 트리 메뉴는 현재처럼 탐색성 중심으로 유지한다.
- 메뉴 hover, active, expand 상태는 배경 톤 변화 중심으로 표현한다.
- 단계가 깊어질수록 들여쓰기와 텍스트 톤 차이로 계층을 만든다.
- 데스크톱 접힘 상태에서도 조작 포인트는 명확해야 한다.

### 6.3 콘텐츠 영역

- 현재 `content` 영역의 여백 구조를 유지하되, 페이지별 최대 폭을 다르게 둘 수 있다.
- 홈/대시보드형 페이지는 카드 그리드 중심
- 위키/정보형 페이지는 읽기 폭 중심
- 게임 페이지는 조작 영역 중심

### 6.4 카드 레이아웃

- 홈 화면과 요약 블록은 현재 구조처럼 카드 기반으로 유지한다.
- 카드 안에서는 선보다 여백과 그룹핑으로 내용 블록을 나눈다.
- 카드 하나에 너무 많은 기능을 담지 않는다.

## 7. 컴포넌트 기준

### 7.1 카드

- 기본적으로 너무 강한 테두리는 줄인다.
- 배경색, 반경, 패딩, 미세한 그림자로 구분한다.
- 라이트 모드에서는 밝은 카드, 다크 모드에서는 차분한 반투명 또는 저채도 표면을 사용한다.
- 홈 카드와 랭킹 카드는 같은 시스템 안에 있는 것처럼 맞춘다.

### 7.2 버튼

- 주요 버튼은 브랜드 컬러를 사용한다.
- 보조 버튼은 유령 버튼 또는 연한 표면 버튼으로 처리한다.
- 관리자/게임/정보 페이지가 섞여 있으므로, 버튼 스타일 수는 많지 않게 유지한다.

권장 분류:

- Primary: 주요 액션
- Secondary: 보조 액션
- Tertiary / Ghost: 덜 중요한 액션

### 7.3 입력창

- 검색, 관리자 입력, 점수 등록 등 다양한 입력 상황을 고려한다.
- 입력창은 깔끔하고 단정해야 하며, 포커스 상태는 명확해야 한다.
- 과한 장식보다 입력 가능 상태와 선택 상태가 잘 보여야 한다.

### 7.4 메뉴 링크

- 상단 메뉴와 사이드 메뉴의 색상 체계는 통일한다.
- 현재처럼 hover, active 상태는 유지하되 조금 더 정돈된 톤으로 다듬는다.
- 클릭 가능한 요소는 텍스트만 바뀌는 것이 아니라 배경과 대비도 함께 반응해야 한다.

### 7.5 로딩 / 스켈레톤 / 상태 UI

- 현재 홈 화면에 있는 스켈레톤 방식은 프로젝트와 잘 맞는다.
- 이후 다른 페이지에도 같은 톤의 스켈레톤 규칙을 확장할 수 있다.
- 에러, 빈 상태, 로딩 상태는 문구 톤과 간격을 통일한다.

## 8. 페이지 유형별 적용 가이드

### 8.1 홈

홈은 이 프로젝트의 디자인 기준점 역할을 한다.

- 큰 제목이나 요약 수치로 첫인상을 만든다.
- 날씨, 랭킹 같은 정보는 카드형 레이아웃으로 정리한다.
- 숫자와 상태 정보는 강조하되, 전체는 과밀하지 않게 유지한다.

### 8.2 게임

게임 화면은 장식보다 조작감과 반응성이 더 중요하다.

- 점수, 상태, 버튼은 빠르게 읽혀야 한다.
- 게임별 개성은 허용하되, 공통 UI는 같은 시스템으로 묶는다.
- 랭킹, 재시작, 입력, 결과 영역은 통일된 카드/패널 언어를 사용한다.

### 8.3 정보 / 위키

- 긴 텍스트를 읽기 쉬운 폭으로 제한한다.
- 제목, 소제목, 본문, 보조 문구 위계를 분명히 한다.
- 메뉴 탐색과 본문 읽기가 충돌하지 않게 한다.

### 8.4 관리자

- 가장 화려할 필요는 없고, 가장 명확해야 한다.
- CRUD, 목록, 상태 표시, 입력 폼은 안정성과 가독성이 우선이다.
- 색상 강조는 상태 전달에만 제한적으로 사용한다.

## 9. 실제 프로젝트에 적용할 때의 우선순위

이 문서는 현재 프로젝트를 전면 재디자인하라는 뜻이 아니다.
아래 순서로 점진 적용하는 것이 맞다.

### 1단계

- 전역 색상 토큰 정리
- 상단 바 / 사이드바 / 콘텐츠 배경 톤 정리
- 카드 경계선 완화

### 2단계

- 홈 카드, 랭킹 카드, 상태 UI 통일
- 버튼과 입력창 스타일 정리
- 라이트/다크 모드 대비 보정

### 3단계

- 게임/정보/관리자 페이지별 세부 가이드 확장
- 숫자/메타 정보용 타이포 포인트 적용
- 모달, 테이블, 로딩 UI 공통화

## 10. Do / Don't

### Do

- 현재 프로젝트 구조를 존중하면서 개선한다.
- 한글 가독성을 최우선으로 둔다.
- 라이트/다크 모드 모두에서 자연스럽게 보이게 한다.
- 카드와 패널은 표면 차이와 여백으로 정리한다.
- 홈, 게임, 정보, 관리자 페이지가 한 서비스처럼 느껴지게 한다.

### Don't

- 모든 화면에 과한 유리 효과나 과한 블러를 남발하지 않는다.
- 다크 모드를 네온 위주의 과장된 스타일로 몰아가지 않는다.
- 위키/관리자 화면까지 동일한 전시형 레이아웃으로 강제하지 않는다.
- 모든 요소에 강한 보더를 넣지 않는다.
- 디자인 개성을 이유로 사용성을 희생하지 않는다.

---

## 11. 최종 정리

이 프로젝트의 디자인은 "멋있어 보이는 한 장의 시안"이 아니라, 여러 성격의 페이지를 하나의 서비스처럼 묶는 시스템이어야 한다.

즉, 목표는 다음과 같다.

- 홈은 조금 더 인상적으로
- 정보는 더 읽기 쉽게
- 게임은 더 명확하게
- 관리는 더 안정적으로

그리고 이 모든 화면이 함께 있을 때도 어색하지 않게 만드는 것이 이 문서의 목적이다.
